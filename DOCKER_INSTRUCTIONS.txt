STUDIO ARCHITECT v3.0 - DOCKER DEPLOYMENT GUIDE
===============================================

This project is configured for a high-performance, containerized deployment using Docker and Docker Compose. It utilizes a multi-stage Dockerfile to build a minimal standalone Next.js production image and runs it behind an NGINX reverse proxy.

---

1. ENVIRONMENT CONFIGURATION
Before building the container, you must ensure your environment variables are correctly configured. Docker Compose requires an `.env` file in the root directory.

Ensure your `.env` file contains the following critical keys:

# Frontend build variables
NEXT_PUBLIC_SUPABASE_URL="your-supabase-project-url"
NEXT_PUBLIC_SUPABASE_ANON_KEY="your-supabase-anon-key"
NEXT_PUBLIC_SITE_URL="your-production-url"

# Backend server variables
SUPABASE_SERVICE_ROLE_KEY="your-supabase-service-role-key"
OPENROUTER_API_KEY="your-openrouter-key"
DEEPSEEK_API_KEY="your-deepseek-key"

*(Note: The `NEXT_PUBLIC` variables must be present during the build stage as they are compiled into the standalone runner!)*

---

2. RUNNING THE CONTAINERS
Once your `.env` file is ready, you can build and launch the environment using Docker Compose.

To build the images and run the containers in detached (background) mode:
$ docker-compose up --build -d

To follow the logs and monitor the server booting up:
$ docker-compose logs -f

---

3. ARCHITECTURAL OVERVIEW
- App Container (Next.js): Runs the standalone Next.js server on internal port 3000. It runs as a secure non-root `nextjs` user. It also dynamically spawns temporary `/tmp` preview workspaces.
- NGINX Container: Placed in front of the application as a high-performance reverse proxy. It exposes ports 80 and 443. It is configured for SSL via Let's Encrypt and handles all project previews via path-based routing (/preview/[port]/) on the main HTTPS port.

---

4. MANAGING THE WORKER
The system now includes an independent 'worker' service that handles all production builds. It is decoupled from the main dashboard, meaning your deployed websites will remain online even if the dashboard is stopped.

To monitor the worker's status and logs:
$ docker-compose logs -f worker

---

5. IDENTITY & AUTHENTICATION
The worker uses 'Identity Sharing' from your host VM:
- It automatically uses your VM's SSH keys and Git configuration.
- To use a different identity, update the volume mounts in docker-compose.yml.

---

6. STOPPING THE CONTAINERS
To gracefully stop the environment without destroying the containers:
$ docker-compose stop

To completely tear down the containers and remove the custom network:
$ docker-compose down

---

7. MAINTENANCE & CLEANUP (NUKE & REBUILD)
If you encounter disk space issues (common on 30GB VMs) or need a 100% clean start:

# A. Stop everything and wipe project-specific Docker volumes
$ docker-compose down -v

# B. (Optional) Deep clean unused Docker data (images/cache)
$ docker system prune -a --volumes -f

# C. Wipe local temporary build workspaces
$ rm -rf tmp/*
$ rm -rf builds/*

# D. Fresh build and restart
$ docker-compose up --build -d
